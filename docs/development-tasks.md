# PulseRoad 开发进度与后续任务

这份文档用于对齐当前项目进度、核心模块边界和接下来的完善方向。项目已经从早期的「用户 -> 团队 -> 产品」扩展到产品反馈、功能开关、事件队列、Redis 缓存和 Vue 前端工作台。

## 当前业务主线

```text
用户 -> 团队 -> 产品 -> 反馈
用户 -> 团队 -> 产品 -> 功能开关
```

当前版本的目标不是一次性做完整的产品管理平台，而是先跑通一个可维护的反馈管理闭环：

- 用户通过认证进入系统。
- 用户创建和加入团队。
- 团队维护产品。
- 产品接收反馈并跟踪处理状态。
- 产品配置功能开关并支持灰度计算。
- Redis 和 RabbitMQ 支撑限流、缓存和事件流转。
- Vue 前端提供基础工作台入口。

## 已完成核心模块

### 1. 基础设施与工程骨架

入口：

- `cmd/api`：HTTP API 服务。
- `cmd/migrate`：Gorm 自动迁移入口。
- `cmd/worker`：RabbitMQ 业务事件消费者。

能力：

- Go module 项目结构。
- Gin HTTP 服务和请求日志中间件。
- MySQL + Gorm 初始化、连接池配置和模型注册迁移。
- 统一响应结构。
- 配置文件和环境变量覆盖。
- Docker Compose 开发环境，包含 MySQL、Redis、RabbitMQ、API、Worker 和前端。
- 后端单元测试覆盖主要 service、handler、publisher、cache 和中间件。

后续需要完善：

- 引入版本化迁移工具，替代纯 `AutoMigrate` 的生产发布方式。
- 补充健康检查的依赖探测，例如 MySQL、Redis、RabbitMQ 状态。
- 增加结构化日志字段，统一 request ID、user ID、team ID 等排查信息。

### 2. 用户认证与登录保护

接口：

```http
POST /api/auth/register
POST /api/auth/login
GET  /api/auth/me
```

能力：

- 用户注册和登录。
- bcrypt 密码哈希。
- JWT 登录态签发与解析。
- `Authorization: Bearer <token>` 保护接口。
- 当前用户查询。
- Redis 登录失败次数限制，降低暴力破解风险。

后续需要完善：

- 增加刷新 Token、退出登录和 Token 黑名单机制。
- 完善密码强度策略、邮箱唯一性校验提示和账号锁定策略。
- 区分认证失败、限流失败和账号状态异常的可观测指标。

### 3. 团队与成员权限

接口：

```http
POST /api/teams
GET  /api/teams
GET  /api/teams/:id
```

能力：

- 用户创建团队。
- 创建者自动成为团队 `owner`。
- 用户查看自己加入的团队。
- 团队详情按成员身份授权。
- `team_members(team_id, user_id)` 有唯一约束。
- owner 可以邀请已注册用户加入团队。
- 被邀请用户可以查看并接受自己的待处理邀请。
- owner 可以查看成员、调整 `owner/member` 角色、移除成员。
- 系统会阻止降级或移除最后一个 owner。

后续需要完善：

- 增加邀请过期、撤销邀请和重新发送邀请。
- 如业务复杂度上升，再扩展 `admin`、`viewer` 等角色。
- 增加团队归档或删除能力，并定义关联产品、反馈和开关的处理方式。

### 4. 产品管理

接口：

```http
POST /api/teams/:team_id/products
GET  /api/teams/:team_id/products
GET  /api/products/:id
```

能力：

- 团队成员可以在团队下创建产品。
- 产品必须归属于团队。
- 团队成员可以查看团队产品列表和产品详情。
- 非团队成员不能创建或查看团队产品。
- 产品详情摘要聚合反馈、评论、投票和功能开关数量。

后续需要完善：

- 增加产品编辑、归档和删除。
- 增加产品标识、描述、负责人等管理字段。
- 当数据量变大时，为产品摘要增加缓存或异步统计。

### 5. 产品反馈

接口：

```http
POST  /api/products/:id/feedback
GET   /api/products/:id/feedback
GET   /api/feedback/:id
PATCH /api/feedback/:id/status
```

能力：

- 团队成员可以在产品下创建反馈。
- 支持查看产品反馈列表和反馈详情。
- 反馈状态支持 `open`、`resolved`。
- 创建和查看反馈都复用产品成员权限校验。
- 反馈列表支持状态筛选和分页。
- 反馈支持评论。
- 反馈支持每个用户一票的投票和取消投票。
- 创建反馈后发布 `feedback.created` 事件到 RabbitMQ。
- 前端产品详情页已经接入反馈列表、筛选、分页、创建、状态更新、评论和投票。

后续需要完善：

- 增加反馈优先级、类型、标签、附件和来源字段。
- 支持订阅、反馈合并和更完整的动态时间线。
- 扩展状态流转，例如 `open`、`planned`、`in_progress`、`resolved`、`closed`。
- 支持列表排序和搜索。
- 将关键反馈事件沉淀为审计日志或动态时间线。

### 6. 功能开关与灰度计算

接口：

```http
POST  /api/products/:id/flags
GET   /api/products/:id/flags
GET   /api/flags/:id
PATCH /api/flags/:id
PATCH /api/flags/:id/toggle
POST  /api/flags/evaluate
```

能力：

- 团队成员可以在产品下创建功能开关。
- 开关按 `product_id + environment + key` 保持唯一。
- 支持 `development` 默认环境。
- 支持开关开启/关闭和 0 到 100 的灰度比例。
- 通过 `user_key` 做稳定哈希，返回灰度命中结果。
- 使用 Redis 缓存开关读取结果。
- 开关创建、更新、切换后清理缓存并发布 `flag.changed` 事件。
- 前端产品详情页已经接入功能开关列表、创建、编辑、启停和计算。

后续需要完善：

- 增加按用户、组织、地区、版本等条件的规则引擎。
- 增加 SDK 或轻量客户端接口，供业务服务直接拉取和评估开关。
- 支持多环境配置复制、审批发布和回滚。
- 增加开关变更历史、命中统计和过期开关清理提醒。
- 评估接口可考虑提供无需团队后台登录的服务端鉴权方式。

## 已集成基础设施模块

### Redis

当前用途：

- 登录失败次数限制。
- 功能开关缓存。

后续需要完善：

- 统一 Redis key 命名规范和 TTL 策略。
- 增加缓存降级策略，避免 Redis 短暂故障直接影响核心流程。
- 评估是否增加反馈热榜、产品摘要缓存或接口限流。

### RabbitMQ

当前用途：

- 反馈创建事件发布与消费。
- 功能开关变更事件发布与消费。
- Worker 消费后记录业务事件日志。

后续需要完善：

- 增加重试、死信队列和消费幂等。
- 将事件处理从日志输出扩展为通知、审计、统计等实际业务动作。
- 统一事件版本、事件 schema 和兼容策略。

## 暂未实现但值得规划的模块

这些模块不建议立即全部展开，应在反馈和功能开关闭环稳定后按优先级推进：

1. 评论与投票：让团队成员围绕反馈补充上下文，并用投票辅助排序。
2. 反馈转需求：把高价值反馈沉淀为产品需求，建立反馈与需求的引用关系。
3. 路线图：按产品和时间维度展示计划、进行中、已发布能力。
4. 发布日志：将已完成需求或功能开关发布结果整理为对外变更记录。
5. 通知：基于 RabbitMQ 事件向团队成员发送站内通知或邮件。
6. 审计日志：记录关键资源的创建、更新、删除和权限变化。
7. 数据看板：统计反馈趋势、处理效率、开关命中和活跃产品。

## 当前维护建议

- 新增业务模块时继续沿用 `model -> repository -> service -> handler -> router` 分层。
- 涉及团队、产品、反馈和功能开关的数据访问必须先走成员权限校验。
- Redis 和 RabbitMQ 依赖应保持可替换接口，方便测试和本地降级。
- 每个模块至少覆盖 service 层核心规则和 handler 层主要 HTTP 行为。
- 前端新增页面时同步补齐 API 封装、类型定义和关键交互测试。
